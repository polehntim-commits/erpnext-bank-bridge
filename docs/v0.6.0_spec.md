# Bank Bridge v0.6.0 ‚Äî MCP server (AI-operable surface)

**Status:** spec ‚Äî awaiting approval
**Author:** drafted 2026-07-23
**Scope estimate:** 2‚Äì3 days focused build for v0.6.0 (Bank Bridge MCP only).
The ERPNext MCP (v0.6.1) and the orchestrator (v0.6.2) are scoped here but are
separate, later builds.

---

## 1. Why this exists

Every diagnostic-and-fix cycle from v0.4.42 ‚Üí v0.5.7 paid the same tax. To read
Bank Bridge's live state ‚Äî an anchor variance, a merchant's twin count, a
statement's match status ‚Äî the loop was: write a `docker exec ‚Ä¶ python -c "‚Ä¶"`
snippet, hand it to Tim, Tim pastes it into his Umbrel SSH session, pastes the
output back. High latency, high transcription-error rate, no direct feedback.
Every fix was gated on that relay.

This is squarely against the mission. Bank Bridge is being built to be
**handoff-ready family-office infrastructure** ‚Äî the thing a bookkeeper (or an
AI acting for one) opens and finds the work already done and inspectable. A
system whose state can only be reached by a human relaying shell commands is not
handoff-ready. Every future Bank Bridge diagnostic and every safe fix should be
a **single AI-callable tool**, not a shell snippet.

v0.6.0 delivers that surface for Bank Bridge. v0.6.1 delivers the matching
surface for ERPNext, so the two can be **cross-verified** in one action.

## 2. The two-server plan

- **v0.6.0 ‚Äî Bank Bridge MCP server (this release).** An MCP endpoint bolted
  onto the existing Flask app at `/mcp`. Read-only tools ship immediately;
  mutating tools ship behind per-tool kill switches that default OFF.
- **v0.6.1 ‚Äî ERPNext MCP server (follow-up, separate codebase).** Tim
  specifically called out ERPNext, above all for reconciliation
  cross-verification. ERPNext is Frappe, not this Flask app, so its MCP lives
  elsewhere ‚Äî a Frappe custom app or a standalone service wrapping the ERPNext
  REST API (¬ß14). Not built in v0.6.0; scoped here so the arc is visible.
- **v0.6.2 ‚Äî orchestrator (later).** A third MCP that has the Bank Bridge and
  ERPNext MCPs as *clients*, so a single cross-system question resolves without
  the caller wiring two servers together (¬ß14).

## 3. Cross-verification ‚Äî the point of both servers

The concrete goal that justifies two MCPs is a question like:

> "Bank Bridge says $0.00 variance on ‚Ä¢‚Ä¢6030 for May 2026. Does ERPNext's Bank
> Statement doctype agree, via the v0.5.0 status push?"

Today that is two SSH sessions and manual comparison. With both MCPs it is one
AI action:

1. Bank Bridge MCP `get_reconciliation_status(account_mask="6030",
   period_start="2026-05-01", period_end="2026-05-31")` ‚Üí the anchor row:
   `variance: 0.0`, `anchored_closing`, `transaction_sum`, dates.
2. ERPNext MCP `get_bank_bridge_status(statement_name=‚Ä¶)` ‚Üí the Bank Statement
   doctype's pushed fields: `bank_bridge_reconciled`, `bank_bridge_variance`,
   `bank_bridge_reason`.
3. The orchestrating AI compares the two numbers and reports agreement or drift.

v0.6.0 ships step 1's tool. v0.6.1 ships step 2's. v0.6.2 makes step 3 a single
tool rather than the caller's glue.

## 4. MCP protocol choice

The MCP wire protocol is **JSON-RPC 2.0**. Two transports exist: stdio (for
subprocess servers) and Streamable HTTP (a long-running HTTP server, optionally
using Server-Sent Events for streaming). Bank Bridge is already a long-running
Flask app, so **HTTP fits** ‚Äî mount the endpoint at `/mcp`.

Implementation note: Bank Bridge is synchronous Flask/WSGI, while the official
`mcp` SDK's *server* runtime is asyncio/ASGI. Bolting the SDK's server loop into
WSGI is fragile. So v0.6.0 implements the JSON-RPC methods an MCP client calls ‚Äî
`initialize`, `tools/list`, `tools/call`, `ping` ‚Äî **directly on the `/mcp`
route**. That is exactly the request/response surface of the Streamable-HTTP
transport, so any MCP client speaks to it unchanged. SSE streaming of partial
tool output is a non-goal for v0.6.0 (¬ß13); every tool here returns a single
JSON result, which the request/response path handles without SSE.

The ERPNext MCP (v0.6.1) makes its own transport decision ‚Äî most likely the
same HTTP+JSON-RPC surface, either inside a Frappe custom app (which is also
long-running) or a standalone wrapper service (¬ß14).

## 5. Auth model

- **Bearer token.** Every `/mcp` request must carry `Authorization: Bearer
  <token>`. The token is read from the environment variable
  `BB_MCP_AUTH_TOKEN` (an Umbrel env override). It is never persisted to disk or
  the database, so a leaked `DATA_DIR` or DB dump cannot expose it.
- **Absent token ‚Üí 404.** If `BB_MCP_AUTH_TOKEN` is unset, `/mcp` returns 404 ‚Äî
  the feature does not exist. A token-less install exposes no AI surface at all.
- **Bad/missing bearer ‚Üí 401.**
- **LAN-only.** The endpoint is reachable only on the Umbrel LAN. It is
  deliberately **not** wired into the `/bankbridge/` Tailscale-Funnel prefix;
  nothing here is meant to be publicly reachable (¬ß13).
- **Path prefix.** Per the multi-app convention in CLAUDE.md, `/mcp` is a
  Bank-Bridge-scoped prefix, so it never collides with another Umbrel app
  sharing the tailnet host.

Two independent gates therefore protect the surface: the token decides *whether
an AI can talk to Bank Bridge at all* (an ops/network decision, hence env), and
the per-tool kill switches (¬ß8) decide *what it may change* (a per-capability
trust decision, hence a settings file the operator edits at `/admin/mcp`).

## 6. Tool catalog ‚Äî Bank Bridge MCP v0.6.0

Fourteen tools: seven read-only (always available) and seven mutating (each
behind a kill switch that defaults OFF). Every tool call, allowed or blocked, is
written to `AiActionLog` (¬ß8). Every tool carries a `description` explaining what
it does and when to use it ‚Äî that text is how the AI picks the right tool from
`tools/list`.

### Read-only tools (ship immediately)

**`get_reconciliation_status`** ‚Äî the statement-anchor chain for one account.
- args: `account_mask` (string, required), `period_start` (string, ISO date,
  optional), `period_end` (string, ISO date, optional)
- returns: `{account_mask, account_id, summary:{variance, unexplained, gaps,
  periods}, anchors:[{period_start, period_end, anchored_opening,
  anchored_closing, transaction_sum, computed_closing, variance, variance_reason,
  chain_gap_from_prior, parser_version}]}`

**`get_account_topology`** ‚Äî every Plaid account as `/admin/accounts` sees it.
- args: none
- returns: `{count, accounts:[{mask, name, type, subtype, owning_company,
  paired_account_mask, superseded_by_account_id, sync_enabled, import_status,
  has_anchor_chain}]}`

**`get_variance_breakdown`** ‚Äî every row contributing to a period's variance.
- args: `account_mask` (string, required), `period` (string `YYYY-MM`, required)
- returns: `{account_mask, period, anchor, bank_transactions:[{date, plaid_date,
  amount, name, match_status, source}], statement_transactions:[{posted_date,
  amount, plaid_convention_amount, description, section, match_status}]}`

**`list_unreconciled_statements`** ‚Äî every period whose |variance| exceeds
tolerance, worst first.
- args: none
- returns: `{count, tolerance, unreconciled:[anchor dict + account_mask]}`

**`list_rules`** ‚Äî categorization rules.
- args: `active_only` (bool, default true)
- returns: `{count, rules:[{id, name, priority, active, match_type, match_value,
  offset_account, bb_internal_tag, applies_to_company}]}`

**`list_unmatched_statement_transactions`** ‚Äî `no_match` statement lines.
- args: `account_mask` (string, required), `period` (string `YYYY-MM`, optional)
- returns: `{account_mask, count, unmatched:[{posted_date, amount, description,
  section}]}`

**`get_advisory_agreement_summary`** ‚Äî advisory dashboard data.
- args: `agreement_id` (int, optional ‚Äî omit for a list)
- returns: the `advisory.dashboard()` dict, or `{count, agreements:[{id, name,
  aum}]}`

### Mutating tools (each behind a kill switch, all default OFF)

**`create_rule`** ‚Äî args `match_type`, `match_value` (required), `offset_account`,
`tag`, `name`; returns the created rule dict.
**`set_variance_tag`** ‚Äî args `anchor_id` (int), `reason` (required); writes
`StatementAnchor.variance_reason`.
**`trigger_reparse`** ‚Äî no args; runs `reparse_stored()`; returns its stats.
**`rebuild_anchors`** ‚Äî arg `account_mask` (optional, else all); runs
`rebuild_statement_anchors()`.
**`pair_accounts`** ‚Äî args `brokerage_mask`, `cash_services_mask` (required);
sets `paired_account_id`.
**`enable_je_posting`** / **`disable_je_posting`** ‚Äî arg `item_id` (required);
flips the Phase-D `PlaidItem.invest_je_posting_enabled` switch.

A mutating tool whose kill switch is OFF returns an MCP tool error
(`isError: true`) whose text names the switch, and logs the blocked attempt with
`ok=false` ‚Äî a denied mutation is as visible in the audit trail as a successful
one.

## 7. Read-only default, mutation off ‚Äî confirmed

On a fresh install: the seven read-only tools work as soon as
`BB_MCP_AUTH_TOKEN` is set; the seven mutating tools are all OFF and every
attempt to call one is blocked-and-logged until an operator flips its switch at
`/admin/mcp`. Read tools never mutate and are never gated. There is no
configuration in which an AI can change the books without an explicit,
per-capability operator opt-in.

## 8. Audit log

New table **`AiActionLog`** (built by `create_all()` ‚Äî no DDL migration
needed): `id`, `created_at` (indexed), `tool_name` (indexed), `args`
(JSON-encoded text), `result_summary` (text), `caller_ip`, `ok` (bool, indexed).
One append-only row per tool call ‚Äî read AND write, allowed AND blocked ‚Äî never
updated, never deleted, the same discipline as `AuditEvent`.

Admin UI at **`/admin/mcp`** shows: endpoint enabled/disabled, the token masked
(all but last four hidden ‚Äî never the token itself), the kill-switch state per
mutating tool with save-toggle form, a "Test connection" button (exercises
`tools/list` in-process), and the recent `AiActionLog` rows (tool, ok/blocked,
result summary, caller). A `MCP` nav link is added.

`AiActionLog` rows in production contain real arguments and figures ‚Äî this is a
live operational record, deliberately outside the PII grep's scope (¬ß11).

## 9. Testing

The suite will guard:
- **Token missing ‚Üí 404** (feature disabled).
- **No / wrong bearer ‚Üí 401**; valid bearer ‚Üí 200.
- **`tools/list`** returns every tool with a non-empty `description` and an
  object `inputSchema`.
- **A read tool returns correct JSON** (e.g. `get_account_topology` count and
  masks; `get_reconciliation_status` anchor variance).
- **An unknown mask is a tool error, not a 500** (`isError: true`, HTTP 200).
- **A mutating tool is blocked when its kill switch is OFF** (isError, message
  names the switch, no row created) and **allowed when ON** (row created).
- **Every call writes an `AiActionLog` row**; a blocked mutation logs `ok=false`.
- **`/admin/mcp` renders** (route-GET, guarding the `_page` reserved-kwarg trap)
  and the toggle form persists switches.

All fixtures use synthetic masks (`4242` / `4200`) and names (`TESTVENDOR`).

## 10. Deployment

- **New dependency:** `mcp` in `requirements.txt`. The server implements the
  wire protocol directly (¬ß4), so this pin is for client/reference use; nothing
  in the server imports it, and the app boots and the suite runs without the
  wheel present.
- **Migration:** none beyond `create_all()` building `ai_action_log`.
- **Env:** set `BB_MCP_AUTH_TOKEN` as an Umbrel app env override, then restart.
  Until then `/mcp` is 404.
- **Client config on Tim's machine.** After deploy, add the endpoint to Claude's
  MCP client config. On the Umbrel LAN the server is reached at the box's LAN
  address, path `/mcp`, e.g.:

  ```json
  {
    "mcpServers": {
      "bank-bridge": {
        "url": "http://umbrel.local:<port>/mcp",
        "headers": {"Authorization": "Bearer ${BB_MCP_AUTH_TOKEN}"}
      }
    }
  }
  ```

  The exact host/port is whatever the Umbrel app proxy exposes on the LAN; the
  token is the same value set in the app env. Because the endpoint is LAN-only,
  the client machine must be on the tailnet/LAN ‚Äî there is no public URL.

## 11. PII rules

The strict v0.4.49+ rules hold: no real party, merchant, mask, or amount in any
test fixture; synthetic names only; the expanded PII grep (POLEHN, DANELLA,
LensCrafters, ‚Ä¶) stays green over all new code and tests; the `.gitignore`
fixture rules stay in place. The one carve-out is **`AiActionLog` rows in
production**, which hold real arguments and figures by design ‚Äî they are a live
audit record, never fixture-serialized and never checked into the repo.

## 13. Non-goals (explicit)

- **Public-internet exposure ‚Äî never.** LAN-only; not on the Funnel prefix.
- **OAuth / SSO flows.** A single bearer token is sufficient for a LAN,
  single-operator surface. No OAuth in v0.6.x.
- **Rate limiting.** A v0.6.2 concern once there is more than one caller.
- **Streaming (SSE) tool responses.** Every v0.6.0 tool returns one JSON result;
  SSE partial-output streaming is deferred to v0.6.2.
- **Multi-tenant token scopes.** One token, all-or-nothing (mutation still gated
  per-tool by the kill switches). Per-scope tokens are a v0.6.2 concern.
- **Write tools beyond the seven listed.** No account deletion, no ERPNext
  posting, no Plaid re-link from MCP in v0.6.0.

## 14. ERPNext MCP ‚Äî outline for v0.6.1

Not shipping in v0.6.0. Sketched so the arc is reviewable.

**Where it lives ‚Äî two options.**
- *Frappe custom app* (an MCP server inside ERPNext). Pro: direct ORM access to
  doctypes, native permissions, one deploy unit. Con: couples to the Frappe
  runtime and its async story; upgrades touch the ERPNext image.
- *Standalone wrapper service* that calls the ERPNext REST API. Pro: isolated,
  independently deployable, testable without a Frappe instance, same
  Flask-style shape as this server. Con: a second service to run; bounded by
  what REST exposes; re-implements permission checks REST already enforces.

Recommendation to revisit at v0.6.1: the **standalone wrapper** mirrors this
server's shape (Flask + hand-rolled JSON-RPC + bearer/API-key auth) and keeps
ERPNext's image untouched ‚Äî the lower-risk path for a handoff-ready system.

**Tool catalog outline (read-first).**
- `get_bank_statement(name)` ‚Äî one Bank Statement doctype record.
- `list_bank_statements(bank_account, from_date, to_date)`.
- `get_bank_bridge_status(statement_name)` ‚Äî reads the v0.5.0 status-push fields
  (`bank_bridge_reconciled`, `bank_bridge_variance`, `bank_bridge_reason`) ‚Äî the
  ERPNext half of the cross-verification in ¬ß3.
- `list_journal_entries(company, from_date, to_date, account?)`.
- `get_gl_balance(account, date)`.
- `list_customers` / `list_suppliers`.

**Auth model.** Reuse ERPNext's own API keys ‚Äî each user has an `api_key` +
`api_secret`. The wrapper takes those (or a dedicated service user's) and passes
them through, so ERPNext's native permission model is the authorization, not a
second bespoke one.

**Cross-verification tool ‚Üí the orchestrator (v0.6.2).** The single-action
"do Bank Bridge and ERPNext agree?" belongs in a **third** MCP that holds the
Bank Bridge and ERPNext MCPs as clients and exposes e.g.
`verify_reconciliation(account_mask, period)` ‚Äî it calls
`bank_bridge.get_reconciliation_status` and `erpnext.get_bank_bridge_status`,
diffs them, and returns agreement/drift. Keeping the orchestrator separate means
neither single-system MCP has to know about the other.
