# Bank Bridge v0.5.0 — Investment Portfolio Tracker + 5:4 Strategy

**Status:** shipped — v0.5.0 milestone
**Author:** drafted 2026-07-20
**Scope estimate:** 5-6 days focused build (Phase C removed — user has TradingView for market data)

---

## v0.5.0 release notes

v0.5.0 ships the investment pipeline and strategy tracker this spec was written
for (Phases A and B) **and** a reconciliation subsystem that grew out of them —
scope added mid-cycle when real Wells Fargo Advisors statements landed and
showed the account's cash never reconciled on its own. The throughline is Bank
Bridge's core mission: **reduce bookkeeping labour** by gathering and
documenting, so a bookkeeper opens the books and finds the work already done.

What actually shipped, factually:

- **Investment data pipeline (Phase A, v0.4.27–30).** Plaid `investments`
  product, holdings + security-transaction models, 730-day history.
- **5:4 lot tracker + options overlay (Phase B, v0.4.31–37).** Configurable
  strategy parameters, lot matching, diversification trigger.
- **Reconciliation subsystem (v0.4.42–0.5.0, scope added).** A durable,
  ERPNext-independent record of what each account actually held at every
  statement boundary — from the bank's own PDF — and the machinery to make it
  reconcile against the transaction feed. See the Scope status section for the
  itemised list. This is the labour-reduction win: a period that reconciles is
  a period nobody has to reconcile by hand, and ERPNext now says so on the
  Bank Statement record.
- **Phase C skipped** (market-data adapter) — the user has TradingView.
- **Phases D and E** remain planned (v0.5.1 and v0.5.2).

No corrections were ever posted to ERPNext: every reconciliation write is
STATUS describing what Bank Bridge concluded, never a Journal Entry or a
balance rewrite, so the accounts that will move to Orchard Meadow's own ERPNext
instance carry nothing that needs unwinding.

## Why this exists

The user operates a disciplined buy-and-hold trading strategy ("5:4 with 400%
diversification trigger") across a Wells Fargo Advisors brokerage held under
Orchard Meadow, LLC, executing 10-20 trades per month. Current Bank Bridge
support for investment accounts is `balance-only`: the total brokerage value
posts to a Marketable Securities GL leaf, but individual securities, lot cost
basis, and trade history are invisible to the app.

v0.5.0 turns Bank Bridge into a real portfolio tracker for this specific
strategy while keeping the general accounting pipeline intact.

## The strategy this tracker is built around

- **5:4 buy pattern.** Buy 5 units of a security. When it appreciates ~25%,
  sell 4 units, keeping 1 unit as a permanent "runner." Retained shares are
  intended to never be sold under normal conditions.
- **400% diversification trigger.** When a retained lot reaches ≥400% of its
  cost basis (i.e., 5× its purchase price), sell 10% of the retained shares
  to reduce concentration risk. This is a mechanical rule, not discretionary.
- **Options overlay for revenue enhancement.** At the ~25% gain point on a
  position, the user discretionarily sells options (covered calls against
  retained shares, or cash-secured puts against dry powder) to collect
  premium as additional income. Position sizing is calibrated to remain
  fully-collateralized — **no naked options, no margin, no debit spreads.**
  If a call is assigned, the shares delivered come from an existing retained
  lot (or a portion of it). If a put is assigned, the resulting long position
  is treated as a new 5:4 cycle entry.
- **No leverage on the investment side.** The user's leverage sits in the
  farming operation (Orchard Meadow, LLC), not the brokerage. The tracker
  must SURFACE any position that violates this — a short option with
  insufficient underlying shares or cash collateral is a violation to flag,
  not a normal state to accept.
- **All parameters configurable.** The tracker hardcodes nothing — retain
  ratio, profit target, diversification trigger, trigger sell percentage,
  and options-writing eligibility criteria are all editable in a settings
  panel so the same code supports variations.

## Data source decisions

- **Trade history + holdings + prices come from Plaid `investments` product.**
  Enabled at Plaid application level; requested optionally in create_link_token
  as of v0.4.26. Plaid returns `institution_price` and `institution_price_as_of`
  on every holding, refreshed on each sync.
- **No separate market data feed.** Skipped by user request — user gets
  real-time market data via TradingView separately. Bank Bridge shows the
  price Plaid returned + the "as of" timestamp so the user knows how stale it
  is; anything more current is TradingView's job.
- **Cost basis reporting.** Plaid reports Wells Fargo Advisors' cost basis
  per position on `/investments/holdings/get`. Trusted for lot classification
  purposes unless the user manually overrides.

## Deliverable phases

Each phase is independently deployable. User can start using each as it lands.

### Phase A — Investment data pipeline (~2 days)

New Plaid client wrappers:

- `investments_holdings_get(access_token) -> list[dict]` — current positions
  with security_id, ticker, quantity, cost_basis, institution_price,
  institution_price_as_of.
- `investments_transactions_get(access_token, start_date, end_date, cursor)`
  — buys, sells, dividends, splits, transfers with security_id + security
  detail. Cursor-paginated same as transactions_sync.

New database models:

- `Security` — one row per unique security_id. Fields: `security_id` (Plaid
  PK), `ticker`, `name`, `type` (equity/etf/mutual_fund/derivative/etc.),
  `currency`, `cusip`, `isin`, `is_option` (bool derived from Plaid type),
  `option_underlying_ticker`, `option_strike`, `option_expiration`,
  `option_contract_type` (`call` / `put`). Plaid returns option contract
  metadata under the security's `option_contract` sub-object; we flatten
  it here so a query for "all AAPL options" is trivial.
- `SecurityHolding` — current snapshot of a position at a Plaid account.
  Fields: `account_id`, `security_id`, `quantity`, `cost_basis_total`,
  `institution_price`, `institution_price_as_of`, `market_value`,
  `refreshed_at`. Negative quantity = short position (options-writing).
- `SecurityTransaction` — one row per Plaid investment_transaction. Fields:
  `plaid_investment_transaction_id` (PK), `account_id`, `security_id`,
  `date`, `action` (buy/sell/dividend/assignment/expiration/exercise/etc.),
  `quantity`, `price`, `fees`, `amount`, `plaid_type`, `plaid_subtype`,
  `created_at`.

Sync loop additions:

- On every scheduled Plaid sync tick, for each Item with `investments`
  product, call the two new endpoints and upsert results.
- Cursor state stored per-Item (like transactions_sync's next_cursor).

Admin UI pages:

- `/admin/holdings` — current positions across all investment accounts,
  scoped by Company. Columns: ticker, name, quantity, cost basis,
  institution_price (with as_of stamp), market value, unrealized %.
- `/admin/investment_transactions` — trade history, filterable by security /
  action / date range / account.

### Phase B — 5:4 lot tracker + options overlay (~2 days)

New database models:

- `RetainedLot` — a lot the user intends to hold. Fields: `security_id`,
  `account_id`, `purchase_date`, `cost_basis_per_share`, `shares_original`,
  `shares_remaining`, `source_cycle_id` (nullable link to TradedCycle),
  `strategy_tag` (default `5_4`, allows manual `manual` etc.), `notes`,
  `created_at`.
- `TradedCycle` — a matched buy/sell pair. Fields: `security_id`,
  `buy_date`, `buy_qty`, `sell_date`, `sell_qty`, `buy_price`, `sell_price`,
  `realized_pnl`, `retain_ratio` (5:4 = 0.2 default), `cycle_status`
  (open/complete/partial).

Pattern detection engine:

- Analyzes SecurityTransaction rows periodically (nightly + on-demand).
- Groups by security_id + close-in-time buy/sell pairs.
- Configurable matching window (default 180 days between buy and sell).
- If a buy N shares is followed by a sell of ~0.8N (within tolerance) at
  ~1.25× the buy price, classify as a 5:4 cycle.
- Creates a TradedCycle + a RetainedLot for the 20% retained portion.
- Unmatched buys (no offsetting sell yet) → `open` RetainedLot marked
  `not_yet_5_4` for user review.

Strategy settings panel:

- `retain_ratio` (default 0.20 → keep 1 of 5)
- `profit_target_pct` (default 25 → sell at 1.25×)
- `matching_tolerance_qty_pct` (default 5 → 76-84% sell qualifies)
- `matching_tolerance_price_pct` (default 3 → 22-28% profit qualifies)
- `matching_window_days` (default 180)
- `diversification_trigger_pct` (default 400 → 5× cost basis)
- `diversification_sell_pct` (default 10 → sell 10% of retained shares)

Manual overrides: on any RetainedLot or TradedCycle, user can toggle
"not-5:4" or "keep-anyway" to override the detector's classification.

**Options overlay tracking:**

New model `OptionsPosition` — one row per open short (or long) option
contract. Fields:

- `security_id` (link to the option Security row)
- `account_id`
- `contract_type` (`call` / `put`)
- `underlying_security_id` (link back to the equity Security)
- `underlying_ticker`, `strike_price`, `expiration_date`
- `contracts_open` (positive = long, negative = short)
- `premium_received_total` (sum of open-write premiums; positive for shorts)
- `premium_paid_total` (for buy-to-close or long positions)
- `net_premium` (premium_received - premium_paid; unrealized income while open)
- `covered_by_shares` (int — how many underlying shares back this short call)
- `covered_by_cash` (float — how much cash collateral backs this short put)
- `is_naked` (bool — computed, true when covered_by_* is insufficient)
- `status` (`open` / `closed` / `expired` / `assigned` / `exercised`)
- `opened_at`, `closed_at`
- `notes` (user-editable free text)

New model `OptionsIncomeEntry` — one row per premium-generating event
(open a short, close a short at profit, expiration of a short). Fields:

- `options_position_id`
- `date`
- `action` (`sell_to_open` / `buy_to_close` / `expired_worthless` /
  `assigned` / `exercised`)
- `contracts`
- `premium_per_contract`
- `total_premium` (contracts × premium × 100 for standard equity options)
- `realized` (bool — true once the position is fully closed/expired)
- `plaid_investment_transaction_id`

Detection engine (extends the 5:4 detector):

- On every SecurityTransaction ingest, when the Security is an option:
  - `sell_short` on an option → new `OptionsPosition` with contracts_open
    negative, premium_received_total incremented
  - `buy_to_close` on an existing short → decrement contracts_open, update
    premium_paid_total, mark closed when contracts_open reaches 0
  - `expired` → mark position status=`expired`, capture full premium as
    realized income
  - `assigned` (short call) → mark status=`assigned`, generate a
    `sell` SecurityTransaction on the underlying at the strike price (Plaid
    should already report this; the detector reconciles the two events),
    tag the retained-lot's shares_remaining reduction as
    `assigned_via_call`
  - `assigned` (short put) → mark status=`assigned`, generate a `buy`
    SecurityTransaction on the underlying at the strike, start a new 5:4
    cycle on the acquired shares

Collateralization check (safety net for the "no leverage" rule):

- On every OptionsPosition update, compute `is_naked`:
  - Short call is naked if `covered_by_shares` < `abs(contracts_open) *
    100` (options are 100-share contracts)
  - Short put is naked if `covered_by_cash` < `abs(contracts_open) *
    strike_price * 100`
- `covered_by_shares` = shares held long in the same underlying at time of
  writing the call
- `covered_by_cash` = cash available in the brokerage cash sweep at time of
  writing the put
- If `is_naked` is true, log a warning and flag on the strategy dashboard.
  The user has explicitly stated no naked options — a positive `is_naked`
  is either a data error or a policy violation, either way worth surfacing.

Strategy settings additions (extends Phase B config panel):

- `options_writing_enabled` (default true; false disables the overlay
  detector for users who don't sell options)
- `options_covered_call_min_gain_pct` (default 25 — matches the 5:4 profit
  target; the user only writes calls at ~25% gain per their description)
- `options_cash_secured_put_dry_powder_pct` (default 50 — max % of cash
  sweep that can back short puts, to leave dry powder for actual buys)
- `flag_naked_positions` (default true; alerts on the dashboard)

### Phase C — Strategy dashboard (~1 day)

New page `/admin/strategy`:

**Retained lots panel** — all lots sorted by unrealized gain % descending:
- Ticker, security name, purchase date, cost basis/share, shares remaining,
  institution price (with as_of stamp), unrealized gain %, days held.
- Row-level flags: red if ≥400%, amber if ≥350%, green otherwise.

**Diversification triggers panel** — lots at ≥400%:
- Ticker, gain %, retained shares, computed "sell 10% = X shares" figure,
  "book realized capital gain of Y" estimate.
- Marks lots the user has already actioned (via manual "acknowledged" toggle)
  so the trigger doesn't nag after decision made.

**Approaching triggers panel** — lots at ≥350% and <400%:
- Heads-up section so nothing sneaks past.

**Recent cycles panel** — most recent TradedCycles:
- Ticker, buy date, sell date, buy price, sell price, cycle P&L.
- Aggregate realized gain YTD at the top.

**Open options positions panel** — every OptionsPosition with `open` status:
- Underlying ticker, contract (C/P + strike + expiration), contracts open,
  premium received to date, days to expiration, coverage status
  (covered / cash-secured / **NAKED — warning**).
- Sort by days-to-expiration ascending so the next-expiring are visible first.

**Options income panel** — realized options income:
- Aggregate premium collected YTD (across all closed/expired positions)
- Breakdown by underlying ticker
- Recent OptionsIncomeEntry rows: ticker, action, date, contracts, total

**Naked position warnings** — if any OptionsPosition has `is_naked=true`:
- Prominent red banner at top of dashboard
- Lists the specific positions and how much collateral is missing
- Suggests either closing the position or adding collateral

**Portfolio concentration panel** — top 5 tickers by market value:
- Concentration warning if any single ticker >20% of total.
- Shows retained vs traded split per ticker.

**Dry powder panel** — cash across brokerage cash sweep accounts (3158, 3194),
labeled "available for next buy."

### Phase D — ERPNext integration (~1 day)

Investment transaction posting:

- Each SecurityTransaction posts as a Journal Entry with security-level
  detail in the user_remark ("Bought 100 AAPL at $150 = $15,000").
- Buys: DR Marketable Securities (specific security), CR Cash sweep.
- Sells: CR Marketable Securities (cost portion), DR Cash sweep, and if
  P&L is realized, CR Realized Capital Gains OR DR Realized Capital Losses.
- Uses **Specific Identification** for cost basis when a lot is tagged as
  a retained/traded pair via TradedCycle. FIFO fallback for untagged sells.

New GL accounts (create on first use):

- `1320.1 - Marketable Securities - <ticker>` per security, so the balance
  sheet shows per-security holdings without dozens of manually-created
  accounts.
- `4200 - Realized Capital Gains - <company>`
- `4210 - Options Premium Income - <company>` (short options treated as
  ordinary income under §1256 rules if held less than a year and closed
  short; long-term capital gain if held longer per §1234A. Tax nuance
  handled by CPA; account keeps the income line traceable regardless.)
- `5710 - Realized Capital Losses - <company>` (parallel account for losses)
- `5715 - Options Premium Losses - <company>` (for buy-to-close at loss)

Options-specific posting rules:

- **Sell-to-open a covered call:** DR Cash sweep, CR Options Premium Income
  (for the premium received). No inventory move — the shares stay put; a
  contingent liability equal to the strike × contracts is disclosed in
  notes but NOT posted (per GAAP for written options at fair value, or
  historical-cost for closely-held; user's CPA picks).
- **Buy-to-close a short position at profit:** DR Cash (net premium
  retained), CR Options Premium Income for the retained portion.
- **Buy-to-close at loss:** DR Options Premium Losses for the loss portion.
- **Expiration worthless (short):** No cash movement, but reclassify the
  accrued Options Premium Income from "unrealized" to realized at the
  expiration date. In practice, if we always posted the premium at
  sell-to-open, expiration is a no-op accounting-wise except for tagging
  the position status.
- **Assignment on a short call:** treat as a sell of the underlying at
  the strike price. The retained lot associated with the covering shares
  decrements shares_remaining; if the lot fully depletes, close the lot.
- **Assignment on a short put:** treat as a buy of the underlying at
  the strike price, plus release of any cash collateral. Starts a new 5:4
  cycle if the retained ratio is respected.

Unrealized gains: NOT posted. GAAP treats held-for-sale as unrealized until
realized. The strategy dashboard displays them; ERPNext's balance sheet
shows Marketable Securities at cost until a sell moves the value.

Intercompany extension: if a security transfers in-kind between OM and BBT
(rare but possible for family transfers), the intercompany detector should
pair the outbound and inbound as an intercompany asset transfer JE.

## Success criteria

Bank Bridge v0.5.0 ships successfully when the user can:

- See every WF Advisors trade in Bank Bridge tied to a specific security
- See current holdings with Plaid-provided prices + as_of stamps
- Have 5:4 cycles auto-classified from real trade history without manual JE
  entry per cycle
- See which retained lots are at ≥400% with computed diversification actions
- Review realized P&L YTD without leaving Bank Bridge
- Have every trade post cleanly to ERPNext with security-level detail
- Configure strategy parameters to future-proof against changes to the rule set

## Non-goals

- Real-time market prices (user has TradingView for that)
- Futures / crypto (WF Advisors focus is equities/ETFs + covered options)
- Multi-leg options strategies (spreads, iron condors, straddles) — the
  user's overlay is single-leg covered calls and cash-secured puts only.
  Multi-leg detection can be added later if the strategy expands.
- Options greeks / IV / theta tracking (that's TradingView / options
  broker native tools)
- Trade EXECUTION (Bank Bridge is a read-only bookkeeping tool; the user
  and their financial advisor execute trades at WF)
- Tax lot optimization suggestions (out of scope; that's a CPA decision)
- Portfolio rebalancing recommendations (that's TradingView's domain)
- Detecting margin usage automatically — Plaid does report `is_margin`
  on some accounts, but the user has explicitly stated no margin, so the
  tracker treats a `true` value as an alert rather than normal state.

### Phase E — Investment Advisory Agreement Automation (~2-3 days)

Codifies the fee, benchmark, performance, and reporting mechanics of the
Investment Management Agreement between Orchard Meadow, LLC (Client) and
Tim Polehn (Manager) into automated Bank Bridge workflows. The agreement
requires ongoing quarterly reporting, high-water-mark fee accounting, and
adherence documentation — all currently manual. This phase makes them
automatic and audit-defensible.

New database models:

- `AdvisoryAgreement` — one row per active agreement. Fields: name,
  effective_date, client_company, manager_name, managed_account_ids,
  fee_account_id, total_base_fee_rate (default 0.0200), bank_fee_rate
  (default 0.0075), performance_fee_rate (default 0.20),
  hurdle_benchmark (default `us_10y_treasury`), high_water_mark_enabled
  (default true), risk_control_config (JSON with position sizing rules),
  status (active/terminated).
- `DailyAUM` — daily snapshot per agreement. Fields: agreement_id, date,
  total_market_value, fee_accrual_daily, cumulative_fee_accrual_qtd.
- `HighWaterMark` — running record of the AUM peak for performance-fee
  ceiling. Fields: agreement_id, mark_date, mark_value,
  established_by_period (references the quarter that set the mark).
- `HurdleRateSample` — daily 10-year Treasury rate. Populated by Fed
  Reserve or Treasury.gov API. Fields: date, rate_pct, source.
- `PerformanceSnapshot` — quarterly performance calculation for the
  advisory agreement. Fields: agreement_id, quarter_end, opening_aum,
  closing_aum, contributions, withdrawals, gross_return_pct,
  net_return_pct, hurdle_return_pct, excess_return_pct,
  performance_fee_accrued, high_water_mark_at_start,
  high_water_mark_at_end, notes.
- `AdvisoryFeeAccrual` — one row per accrual event. Fields: agreement_id,
  accrual_date, fee_type (`base` / `performance` / `bank`), amount,
  posted_to_erpnext (bool), erpnext_je_id.
- `RiskControlCheck` — daily compliance snapshot. Fields: agreement_id,
  check_date, position_concentrations (JSON), single_position_limit_pct,
  sector_concentration_limit_pct, bitcoin_allocation_pct,
  violations (JSON array of triggered rules).

Fee calculation engine:

- Daily AUM sampler: for every account under the agreement, sum current
  market value from Phase A holdings + cash sweep balances. Record in
  `DailyAUM`.
- Base fee accrual: `daily_aum × total_base_fee_rate / 365`. Split by
  the `bank_fee_rate` and remaining as `manager_base_fee_rate`. Bank fee
  is recorded but NOT auto-posted (WF Advisors deducts it directly);
  manager fee is accrued to `1250-Fee Accrual` payable.
- Quarterly settlement: at quarter-end, aggregate the QTD accrual and
  generate an ERPNext Journal Entry: DR `5850 - Investment Advisory
  Fees - OML`, CR `1253 - Fee Account (6030)`. Mark posted.

Performance fee engine:

- At quarter-end, compute time-weighted return of the managed accounts:
  `(closing_aum − opening_aum − contributions + withdrawals) / opening_aum`
- Compute cumulative Hurdle Rate return for the quarter from
  `HurdleRateSample`.
- Compute excess return = portfolio return − hurdle return.
- Check high-water mark: performance fee applies ONLY if closing_aum >
  previous high-water mark. If underwater, no performance fee accrued
  (recapture prevention).
- Performance fee = excess_return_pct × performance_fee_rate ×
  average_aum_qtd.
- Book as `AdvisoryFeeAccrual` with `fee_type='performance'`; ERPNext
  JE posts annually per the agreement's payment cadence (accrued
  quarterly, paid annually subject to Client approval).

Hurdle rate feed:

- Daily poll to a public data source. Options:
  - Federal Reserve FRED API (`DGS10` series) — free, requires API key
  - Treasury.gov direct XML feed — free, no key
- Stored in `HurdleRateSample` with `source` field.
- Historical backfill script for first-run so quarterly calculations
  can be done for periods pre-dating installation.

Risk controls monitor:

- Daily job checks each SecurityHolding under the agreement:
  - Single-position % of portfolio (violation if > 5-10%, configurable)
  - Sector concentration (requires securitySector lookup, TBD source)
  - Bitcoin allocation (violation if > 5%)
  - New-entry sizing (violation if any single transaction > 2.5% AUM)
- Violations recorded in `RiskControlCheck.violations`.
- Dashboard surface: red banner if any active violations, with
  suggested corrective action.

Quarterly report generator:

- `/admin/advisory/reports/<agreement_id>/<yyyy-Q>` route generates the
  quarterly report the agreement requires. Sections:
  - Holdings snapshot (Phase A data)
  - Transaction log for the quarter (Phase A data)
  - Performance attribution (Phase B TradedCycle detail + Phase D
    aggregate)
  - Adherence to strategy summary (from `RiskControlCheck` compliance
    log for the quarter)
  - Fee summary (base + performance accrued + paid)
  - Portfolio composition vs strategy targets
- Exportable as PDF (via existing PDF generation used for statements) so
  the Client can archive per the agreement's recordkeeping requirements.

Admin UI additions:

- `/admin/advisory` — list of active `AdvisoryAgreement` records, click
  through to per-agreement dashboard
- `/admin/advisory/<id>` — current AUM, quarter-to-date fee accrual,
  YTD performance vs hurdle, high-water mark status, recent risk
  control violations, quarterly report links
- Settings page for editing agreement parameters (fee rates, risk
  control config, benchmark selection)

Personal-side considerations (out of scope for Bank Bridge, but flagged):

- The Manager Base Fee and Performance Fee constitute self-employment
  income to Tim Polehn personally. Bank Bridge is the CLIENT's system;
  Tim's personal income tracking (for Schedule C / SE tax) is
  separate. Phase E generates the summary reports Tim uses to file his
  personal taxes; it doesn't manage his personal books.
- Estimated quarterly SE tax payments should be made by Tim from fees
  received; a separate personal-side spreadsheet or Quicken file
  tracks this.

## Scope status

- **Phase A — ✓ shipped v0.4.27–30.** Investment data pipeline: Plaid
  `investments` product, holdings + security-transaction models, 730-day
  history.
- **Phase B — ✓ shipped v0.4.31–37.** 5:4 lot tracker + options overlay:
  configurable strategy parameters, lot matching, 400% diversification trigger.
- **Phase C — ✓ skipped intentionally.** Separate market-data adapter. The
  user has TradingView; Plaid's `institution_price` + `as_of` stamp is
  sufficient for Bank Bridge's tracking role.
- **Reconciliation subsystem — ✓ shipped v0.4.42–0.5.0 (NEW SCOPE ADDED).**
  Not in the original spec; added when production Wells Fargo Advisors
  statements showed the brokerage cash never reconciled on its own. What it
  contains:
  - **Statement-anchored ledger** — a `StatementAnchor` per period holding what
    the bank asserted the account opened and closed at, independent of ERPNext,
    so the history replays cleanly onto a future Orchard Meadow instance.
  - **Wells Fargo Advisors PDF parser** — extracts the cash-and-sweep balances,
    portfolio value, income, fees and gain/loss from the real SNAPSHOT layout;
    self-heals across parser upgrades via a stamped `parser_version`.
  - **Account pairing + supersede-chain traversal + fingerprint dedupe** — a
    brokerage account and its Brokerage Cash Services companion reconcile as one
    economic account; re-linked accounts whose history is split across two Plaid
    rows are walked whole, and the overlap those re-links duplicate is counted
    once.
  - **Paired-brokerage security-transaction skip** — a paired account's cash is
    entirely on its sweep companion, so its own security transactions are not
    double-counted (verified against 26 months of production data: total
    variance → $0.00 on the account it was breaking).
  - **Rules-driven internal attribution tags** — one categorization rule serves
    both categorization and reconciliation attribution; tags stay
    Bank-Bridge-internal and never reach ERPNext.
  - **ERPNext reconciliation-status push** — three custom fields
    (`bank_bridge_reconciled`, `bank_bridge_variance`, `bank_bridge_reason`) on
    the Bank Statement record so a bookkeeper sees "already reconciled" without
    leaving ERPNext. STATUS only — no Journal Entry, no balance rewrite, nothing
    to unwind.
- **Phase D — ✓ shipped v0.5.1.** Investment transactions posted as Journal
  Entries: buys/sells/fees/dividends/written options with security-level
  detail, per-security Marketable Securities accounts, realized gain/loss on
